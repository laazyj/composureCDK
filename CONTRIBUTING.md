# Contributing to ComposureCDK

Thanks for your interest in ComposureCDK! This guide explains how to get
involved while the project finds its feet.

> [!IMPORTANT]
> ComposureCDK is **pre-release (0.x)** and its architecture is still
> stabilising. Right now the most valuable contribution is **conversation** —
> issues, questions, and design feedback — rather than large pull requests
> against an API that may still change.

## How to contribute

We work **issue-first**. Starting with an issue or discussion lets us agree on
fit and approach before anyone writes code, which avoids wasted effort while the
public API is fluid.

- **Found a bug?** [Open a bug report](https://github.com/laazyj/composureCDK/issues/new).
  Include the package and version, a minimal reproduction, and what you
  expected to happen.
- **Have a feature idea or design question?**
  [Open an issue](https://github.com/laazyj/composureCDK/issues/new) to discuss
  it **before** sending a PR. We'd rather shape the idea together than decline a
  finished PR that doesn't fit the architecture.
- **Small, obvious fixes** — typos, broken links, documentation clarifications,
  clear one-line bug fixes — are welcome as direct pull requests without a prior
  issue.
- **Larger changes** — new components, new packages, API changes — please open
  an issue first. Unsolicited large PRs may be declined or asked to wait while
  the architecture settles. This isn't a reflection on the work; it's about
  keeping the design coherent during 0.x.
- **Security issues** — please **do not** open a public issue. Follow the
  process in [SECURITY.md](SECURITY.md).

## Development setup

Prerequisites: **Node.js >= 20** and npm.

```sh
git clone https://github.com/laazyj/composureCDK.git
cd composureCDK
npm install
```

This is an [nx](https://nx.dev/) monorepo; the publishable packages live under
[`packages/`](packages/). Use `npx nx` to run targets.

```sh
# Build everything
npm run build

# Test everything
npm run test

# Work on a single package
npx nx build @composurecdk/core
npx nx test  @composurecdk/core

# Run the example stacks (synth/unit tests)
npx nx test @composurecdk/examples
```

## Project standards

Please read [AGENTS.md](AGENTS.md) — it's the canonical list of coding standards
and is referenced (not duplicated) here. The highlights:

- **Architecture** — [docs/architecture.md](docs/architecture.md) is the primary
  guide to the library's shape (lifecycle, builders, composition, refs,
  defaults). Decisions that amend or add detail live in
  [docs/adr/](docs/adr/). A change that introduces a new pattern, or reverses an
  existing one, should ship with an ADR.
- **Commits** — use [Conventional Commits](https://www.conventionalcommits.org/)
  (`feat:`, `fix:`, `docs:`, etc.). `CHANGELOG.md` files are generated by the
  release process — **never edit them by hand**; describe behaviour changes in
  the commit message and PR body instead.
- **Dual ESM/CJS** — every publishable package ships dual ESM/CJS via `tshy`
  ([ADR-0007](docs/adr/0007-dual-esm-cjs-publishing.md)). Do not use
  `import.meta` or top-level `await` in `src/`, and use a `Symbol.for(...)`
  brand rather than `instanceof` for cross-realm identity checks. The
  `composurecdk/no-cjs-incompatible-syntax` ESLint rule enforces the first part.
- **Adding a package or example** — follow the checklists in
  [AGENTS.md](AGENTS.md) (stack naming, registration, README rows, smoke tests).

## Before you open a pull request

- Run the full gate locally — it's what CI runs, and a husky `pre-push` hook
  runs it automatically:

  ```sh
  npm run verify
  ```

  This chains build, export checks (`attw` + `publint`), lint, format check, and
  tests.

- Auto-fix lint and formatting where possible:

  ```sh
  npm run lint:fix
  npm run format
  ```

- Add or update tests for the behaviour you change.
- Keep PRs focused — one logical change per PR is easiest to review.
